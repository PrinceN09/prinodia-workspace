# ADR-002 — Use Prisma 5 as the ORM

**Status:** Accepted
**Date:** 2026-06
**Deciders:** GovSphere Engineering Team

---

## Context

GovSphere's API layer needs to interact with a PostgreSQL database in a type-safe, maintainable way. The team writes TypeScript exclusively. The ORM must:
- Produce typed query results that reflect the schema
- Manage schema migrations reliably
- Support complex relational queries (joins, nested includes)
- Be compatible with NestJS and Turborepo monorepo structure
- Provide a good developer experience for a team that may rotate members

## Decision

We will use **Prisma 5** as the ORM, schema migration tool, and database client.

The Prisma schema (`packages/database/prisma/schema.prisma`) is the source of truth for the database schema. No raw SQL DDL is used except in migration files generated by Prisma.

## Alternatives Considered

**TypeORM:**
- Rejected. Decorator-based schema definition diverges from the actual DB schema, leading to subtle drift bugs. Migration system is less reliable than Prisma's. Active maintenance has slowed.

**Drizzle ORM:**
- Considered. Excellent TypeScript inference. However, less mature migration tooling at time of decision, and Prisma's ecosystem (Prisma Studio, broader team familiarity) was decisive.

**Knex.js (query builder only):**
- Rejected. Not an ORM; requires manual type definitions for query results. No migration safety.

**Raw `pg` queries:**
- Rejected. No type safety on query results, no migration management, high maintenance burden.

## Consequences

**Positive:**
- Generated TypeScript client is 100% typed — all model fields, relations, and enums are inferred from the schema
- `prisma migrate dev` tracks schema changes as versioned SQL files
- `prisma generate` regenerates the client after any schema change — type errors caught at compile time
- Prisma Studio provides a visual database editor for development
- `schema.prisma` is readable by non-engineers — easier to review in PRs
- Connection pooling via Prisma Client connection pool (single `PrismaClient` instance per app)

**Negative:**
- Prisma does not support all PostgreSQL features natively (some require `$queryRaw`)
- N+1 queries possible if `include` is used carelessly — requires query review
- Schema migration requires running `prisma migrate dev` locally before push; CI must also run it against test DB

## Rules

1. A single `PrismaService` extends `PrismaClient` and is provided as a global NestJS module
2. No raw `new PrismaClient()` calls outside `PrismaService`
3. All schema changes require a new named migration: `prisma migrate dev --name <description>`
4. Never modify existing migration SQL files after they have been applied to any environment
5. `$queryRaw` is allowed for performance-critical queries but must include a TypeScript return type
